# 03. 시나리오

## 지영의 시나리오

### S1. 아침 출근 — 프로필 설정
- **Context**: 다이어트 결심 후 출근길 지하철에서 앱을 처음 연다
- **Flow**: 키/몸무게/나이/성별/활동량 입력 → TDEE 계산 → "오늘 목표: 1,650kcal" 확인
- **인사이트**: 첫 화면이 프로필 입력이면 진입 장벽. 빠르게 끝나야 함 (5개 필드, 30초 이내)

### S2. 점심 메뉴 선택 — 먹기 전 확인
- **Context**: 회사 근처 식당 앞에서 "뭐 먹지?" 고민
- **Flow**: 앱 열기 → 남은 칼로리 확인 (1,350kcal) → 추천 음식 확인 → "비빔밥 550kcal, 괜찮네" → 메뉴 결정
- **인사이트**: 점심 전 추천 확인이 핵심 사용 흐름. 추천 접근이 1탭 이내여야 함

### S3. 점심 식사 후 — 기록
- **Context**: 비빔밥 먹고 자리로 돌아옴
- **Flow**: 앱 열기 → "비빔밥" 검색 → 선택 → 추가 → 남은 칼로리 800kcal 확인
- **인사이트**: 검색 → 선택 → 추가가 10초 안에 끝나야 지속 가능

### S4. 저녁 간식 충동 — 초과 경고
- **Context**: 저녁 먹고 야식 충동. 남은 칼로리 150kcal
- **Flow**: 치킨 검색 → 850kcal → 추가 → 목표 초과! 빨간색 경고 표시
- **인사이트**: 초과 시 강한 시각적 피드백(빨간색 + 경고 메시지)으로 인지

## 상훈의 시나리오

### S5. 첫 사용 — 프로필 설정
- **Context**: 아내가 보내준 링크로 앱을 처음 연다
- **Flow**: 프로필 입력 화면 → 키/몸무게 입력 → 활동량은 "?" 모르겠음 → 기본값(보통) 선택 → "오늘 목표: 2,100kcal"
- **인사이트**: 활동량 등 모호한 항목에 기본값/설명 필요

### S6. 점심 기록 — 검색 의존
- **Context**: 구내식당에서 제육볶음 먹음
- **Flow**: 앱 열기 → "제육" 검색 → "제육볶음 680kcal" 선택 → 추가 → 남은 칼로리 확인
- **인사이트**: 검색 결과가 한글로, 한눈에 보여야 함. 결과 없으면 당황

### S7. 저녁 뭐 먹지 — 추천 의존
- **Context**: 퇴근 후 집에서 저녁 뭘 먹을지 고민
- **Flow**: 앱 열기 → 남은 칼로리 1,200kcal → 추천 음식 목록 확인 → "잡곡밥 + 된장찌개 = 550kcal, 좋은 선택!" → 결정
- **인사이트**: 추천 목록이 상훈의 메인 화면이나 다름없음

### S8. 검색 실패 — 유사 음식 + 카테고리 선택
- **Context**: 동네 분식집에서 먹은 음식이 정확히 DB에 없음
- **Flow**: "쫄면" 검색 → 정확한 결과 없음 → "비슷한 음식: 비빔면 480kcal, 냉면 430kcal" 유사 음식 제안 → 그래도 없으면 → 카테고리 선택: "면류 1인분 약 500kcal" → 추가
- **인사이트**: 칼로리를 모르는 사용자도 막히지 않아야 함. 유사 음식 제안 → 카테고리 평균 칼로리 순으로 폴백

## 설계 원칙

| 원칙 | 근거 시나리오 | 설명 |
|---|---|---|
| P1. 30초 온보딩 | S1, S5 | 프로필 입력은 5개 필드, 기본값 제공, 30초 이내 완료 |
| P2. 10초 기록 | S3, S6 | 검색 → 선택 → 추가가 3탭 이내 |
| P3. 1탭 추천 접근 | S2, S7 | 메인 화면에서 추천 음식까지 1탭 이내 |
| P4. 강한 초과 경고 | S4 | 칼로리 초과 시 빨간색 + 경고 메시지 |
| P5. 검색 실패 폴백 | S8 | 유사 음식 제안 → 카테고리 평균 칼로리 선택 → 수동 입력 순으로 폴백 |
| P6. 큰 숫자 중심 UI | S5, S7 | 남은 칼로리를 크고 명확하게 표시 |
